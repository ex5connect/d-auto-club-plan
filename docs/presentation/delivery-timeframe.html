Detailed milestones

Frontend and backend work by month

Month 1
Weeks 1–4

Foundation, identity and vehicle profiles

Build reusable foundations first so two apps do not become two unrelated codebases.

Start immediately
Frontend tasks
  • Create User and Seller app bundles in one shared repository
  • Set up navigation, environments, API client, secure storage and feature flags
  • Recreate design tokens and reusable components from PNG references
  • Implement social login shell, profile and address forms
  • Implement My Cars and vehicle selection in User App
  • Set up unit/component tests, linting and beta build pipeline
Backend tasks
  • Initialize Laravel API, MySQL, queues, cache, object storage and CI
  • Define API v1 responses, validation, errors, pagination and OpenAPI
  • Implement social auth, users, tokens, devices, roles and permissions
  • Create seller profile/type/application foundations
  • Create vehicle brand, series, variant, year and user-car models
  • Add logging, audit, monitoring, test factories and seed/import structure
Validation & gate
  • Both apps log in against staging
  • User can create/edit/default a car
  • Seller onboarding draft saves correctly
  • OpenAPI and mobile API client agree
  • CI runs tests on every change

Dependencies: approved social providers, launch markets, initial vehicle dataset.

Month 2
Weeks 5–8

Product catalogue and compatibility

Deliver the spare-parts differentiator and seller listing workflow.

Data source required
Frontend tasks
  • User App: categories, product lists, details, media and seller profile
  • User App: search, filters, sorting, favourites and compatible-car status
  • Seller App: create/edit products, media, category, attributes, price and stock
  • Seller App: vehicle compatibility selection and publication status
  • Add loading, empty, validation, offline and error states
  • Contract-test mobile models against API schema
Backend tasks
  • Products, seller SKUs/offers, categories, attributes and images
  • Price, stock, publication state, ownership and moderation policies
  • Vehicle compatibility mapping, source provenance and correction history
  • Search, filters, sorting, pagination and selected-car matching
  • Wishlist, reviews, seller replies and reports foundations
  • Admin APIs for category, listing and compatibility review
Validation & gate
  • Seller publishes an approved spare part
  • Buyer sees it for the correct default car
  • Unknown compatibility is not shown as compatible
  • Stock and price validation pass
  • Search response meets agreed performance

Dependencies: compatibility granularity, legal dataset/scrape source, approval policy.

Month 3
Weeks 9–13

Chat, orders and first end-to-end journey

Create the shared transaction spine before adding more supporting modules.

Escrow API required
Frontend tasks
  • Shared conversation list/thread, message cache and realtime connection
  • Text, image, product and order cards with retry/read states
  • User App: contact seller and create order entry point
  • Seller App: buyer conversations and order context
  • Initial buyer/seller order lists and status components
  • Push-notification and deep-link handling
Backend tasks
  • Conversations, participants, messages, media and read state
  • Realtime events, offline push, rate limits, block/report and retention
  • Cart/order snapshots, order items and initial state machine
  • Escrow provider interface and transaction/link model
  • Notification templates/events and audit history
  • Used-car and car-plate core APIs if capacity allows
Validation & gate
  • Buyer opens product chat with correct context
  • Messages deliver, retry and mark read
  • Seller receives an order record
  • Unauthorized users cannot read conversations
  • First staging buyer-to-seller slice works

Dependencies: realtime provider/approach, message policy, escrow provider sandbox.

Month 4
Weeks 14–17

Escrow, tracking, refunds and supporting discovery

Complete the operational transaction loop and add remaining user-facing modules.

Business rules required
Frontend tasks
  • Seller sends allowlisted escrow payment-link card
  • User sees link, expiry and provider-confirmed payment status
  • Seller submits courier and tracking; buyer sees tracking card/history
  • Cancellation, refund, return and exception screens in both apps
  • Used cars, plates, workshop, EV charging and community screens
  • Seller address/location and listing flows
Backend tasks
  • Create/record escrow links and verify signed idempotent webhooks
  • Implement paid, expired, cancelled, disputed and refunded states
  • Complete order state machine, stock rules and seller fulfilment
  • Courier/tracking validation, event history and buyer notifications
  • Cancellation, refund, return and dispute records
  • Supporting listing/maps/community APIs and minimum operational admin
Validation & gate
  • Duplicate webhook does not duplicate payment
  • Arbitrary links are not shown as verified payments
  • Paid order can ship, track and complete
  • Cancel/refund/return states stay consistent across apps
  • Admin can resolve basic exceptions

Dependencies: escrow webhook capability, refund/dispute policy, tracking requirements, admin owner.

Month 5
Weeks 18–21

Feature integration, operations and beta

Stop building isolated screens and prove complete journeys.

Feature-complete target
Frontend tasks
  • Cross-app integration and shared status consistency
  • Complete notifications, settings, reviews, reports and history
  • Close empty/error/offline/permission/expired-link states
  • Accessibility, localization and device compatibility review
  • Performance tuning, crash reporting and beta distribution
  • Fix PM beta feedback without expanding scope
Backend tasks
  • Complete admin APIs, approval queues and audit views
  • Escrow exception reconciliation and operational exports
  • Reported chat, user, seller, listing and review moderation
  • Performance indexes, cache strategy and queue reliability
  • Security review, abuse controls and data-retention jobs
  • Staging migration, backup/restore and monitoring validation
Validation & gate
  • Critical journeys pass automated regression
  • Feature-complete beta distributed
  • No unresolved critical authorization defects
  • Escrow/tracking exception handling tested
  • Operations can run MVP without database edits

Gate: feature freeze at end of Month 5 except approved launch blockers.

Month 6
Weeks 22–26

Regression, UAT and production readiness

Protect this month from new feature work.

Launch gate
Frontend tasks
  • Device/OS regression for both apps
  • Social login, push, deep links, offline and upgrade tests
  • Performance, accessibility and localization fixes
  • App privacy declarations, signing and store submission
  • Crash-free beta monitoring and launch-blocker fixes
  • Versioning, forced/optional upgrade behavior and release notes
Backend tasks
  • Load, concurrency, webhook, permission and upload testing
  • Escrow reconciliation and dispute/refund rehearsal
  • Production migration, scheduled jobs and queue recovery
  • Backup/restore, alert, incident and rollback rehearsal
  • API documentation, runbooks and support handoff
  • Only launch-blocking defects after code freeze
Validation & gate
  • PM signs off actor-based UAT
  • Infra signs off deploy/rollback/recovery
  • Escrow provider staging reconciliation passes
  • Critical and high launch blockers closed
  • Management makes go/no-go decision

Rule: failed readiness gate moves the launch; it does not remove security or finance testing.

Estimate and contingency

Which timeframe should management approve?

Controlled release
6 months

Medium confidence only when scope locks, providers are ready, minimum admin is accepted and Month 6 is protected.

Likely deferral order

  1. Advanced analytics
  2. Community enhancements
  3. Workshop/EV enhancements
  4. Car-plate enhancements
  5. Used-car enhancements
Recommended estimate
8–10 months

Most credible for all visible scope, complete admin operations, edge states, production quality and reasonable contingency.

Extra time supports

  • Full supporting modules
  • Robust disputes/refunds
  • Device and localization polish
  • Operational admin and reporting
  • Safer rollout and defect recovery
High-risk attempt
6 months full

Low confidence without added QA/admin capacity, immediate P0 decisions and unusually low requirement/provider delay.

Failure signals

  • Escrow undecided after Month 1
  • Compatibility source not ready
  • Admin starts in Month 5
  • Feature work enters Month 6
  • No daily regression owner
