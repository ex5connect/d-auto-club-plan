07 · Escrow, chat & logistics

Do not treat arbitrary links as payments

Provider capabilities open

Recommended escrow sequence

  1. Platform or seller creates a provider transaction.
  2. Backend stores provider ID, amount, order, seller, and expiry.
  3. Seller sends an allowlisted link card through chat.
  4. Buyer opens the provider page.
  5. Signed webhook confirms paid / expired / cancelled / disputed / refunded.
  6. Backend updates both apps and the audit trail.

Seller-managed logistics

  1. Paid order enters To Ship.
  2. Seller chooses courier and enters tracking.
  3. Backend validates required fields and records history.
  4. Buyer receives a tracking card and notification.
  5. Seller and admin handle delay, invalid tracking, return, and refund exceptions.

Meeting decision: manual tracking only, or courier API verification?

19 · Feasibility verdict

Is six months doable?

YES*

for a controlled release with explicit conditions

Six months works when

  • MVP and admin scope lock immediately.
  • Escrow provider exposes usable APIs and signed webhooks.
  • Both apps share packages and contracts.
  • Compatibility source is legal and reliable.
  • Testing runs continuously; Month 6 is protected.

Six months fails when

  • “Ideally all” becomes an unprioritized fixed commitment.
  • Admin, refund, seller, or data rules arrive late.
  • Chat/payment links are built without provider certainty.
  • No one owns daily regression and device testing.
  • Feature work continues through the launch window.

24 · Recommendation

D

Treat six months as a gated target for a controlled release—not an unconditional promise for every screen.
